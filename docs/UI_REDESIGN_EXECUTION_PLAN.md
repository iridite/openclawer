# OC-Deploy UI 重构执行手册（V2）

## 0. 文档定位

本文件是对 V1 方案的“审查 + 优化”版本，目标是让后续 AI 可以按部就班执行，不再依赖口头解释。

适用范围：

1. `app/ui/management.html`
2. `app/ui/assets/management.js`
3. `app/ui/assets/management.css`
4. 必要时 `app/ui/config`（应用名称显示）

约束条件：

1. 不引入前端框架和构建工具。
2. 不改后端 API 协议（除非修复明确 bug）。
3. 优先保持已有稳定功能。

---

## 1. 审查结论（对 V1 方案）

本节以“审查者 + 人机功效（HCI）”视角给出问题清单，按严重级排序。

### 1.1 P0（必须先修）

1. 缺少可量化目标。
说明：V1 只有方向，没有可测指标，执行后无法判断“是否真的更好”。

2. 信息架构存在潜在分裂风险。
说明：将“系统与日志”与“原生控制面板”拆成两个低频 tab，可能增加导航切换成本。

3. 缺少“关键任务流优先级”。
说明：模型与渠道的增删改是高频任务，但文档未显式定义“首要优化对象与路径预算”。

### 1.2 P1（应修）

1. 可访问性要求过于泛化。
说明：V1 提到了 aria/focus，但未给 WCAG 级别与检查阈值。

2. 视觉系统规范不够工程化。
说明：有颜色建议，但缺少间距、层级、状态反馈和图标规则的统一 token。

3. 缺少发布/回滚与灰度策略。
说明：UI 大改若触发操作误判，需要快速止损路径。

### 1.3 P2（建议修）

1. 缺少“决策记录”。
说明：后续 AI 不知道为什么这么设计，容易反复推翻。

2. 缺少“失败场景”的可用性验证。
说明：例如保存失败、网络超时、重启网关失败的引导体验未被纳入验收。

---

## 2. 优化后目标（可测）

本次 UI 改造的成败由以下指标判断（至少满足 80%）：

1. 关键任务平均点击步数减少 20% 以上。
2. 新手完成“添加模型并设为激活”的时间少于 90 秒。
3. 关键表单错误率（前端校验失败）降低 30% 以上。
4. 用户误入“原生控制面板”的语义混淆反馈为 0（Beta 阶段人工验证）。
5. 移动端（375px）无横向滚动、无主要功能遮挡。

## 3. 用户与任务模型（HCI 基线）

### 3.1 主要用户画像

1. NAS 普通用户（非开发者）
目标：快速接入模型与渠道，能跑起来。
特征：怕复杂 JSON，偏好表单。

2. 高级用户（开发/运维）
目标：精细控制配置，快速排错。
特征：接受 JSON 编辑器与日志路径。

### 3.2 Top 8 核心任务（按频次和风险排序）

1. 查看网关状态并重启
2. 添加新模型
3. 设置激活模型
4. 编辑模型
5. 添加消息渠道
6. 编辑消息渠道
7. 保存 JSON 配置
8. 跳转到原生控制面板（dashboard）

原则：高频任务要“少跳转、低记忆负担、低误触”。

---

## 4. 优化后信息架构（V2 推荐）

## 4.1 顶层 Tabs（推荐 5 个）

1. `概览`
2. `模型`
3. `渠道`
4. `配置`
5. `系统`

说明：将“版本、日志、原生控制面板入口”统一放入 `系统`，减少低频 tab 数量，降低导航切换成本。

## 4.2 与 V1 的差异及理由

1. 由 6 tabs 收敛到 5 tabs。
理由：控制选择数量（Hick 定律），让用户更快定位。

2. 原生控制面板不再独立成顶层 tab。
理由：它本质是“外部入口动作”，并非独立业务域。

3. 高风险操作集中在“概览 + 配置 + 系统”。
理由：便于形成操作心智模型（运行态、配置态、维护态）。

## 4.3 每个 Tab 的职责

### A. 概览

1. 展示：Gateway/Proxy 状态、版本、CPU/内存、配置摘要。
2. 操作：启动、停止、重启 Gateway。
3. 入口：进入原生控制面板（dashboard）的快捷按钮。

### B. 模型

1. 展示：模型列表、激活模型标识。
2. 操作：添加新模型、快速添加、编辑、删除、点击设为当前模型。
3. 要求：添加后不抢占当前激活模型（保持已有修复行为）。

### C. 渠道

1. 展示：渠道卡片列表。
2. 操作：添加消息渠道、编辑、删除。
3. 说明：仅提供最简化字段，进阶项去 JSON 编辑器。

### D. 配置

1. 展示：编辑器、校验状态、模板。
2. 操作：重新加载、复制配置、恢复原始配置、保存配置。
3. 要求：校验失败仅警告不阻止保存（保持现有策略）。

### E. 系统

1. 展示：版本信息、更新状态、日志、常用路径。
2. 入口：原生控制面板地址与 token 信息（强调语义区分）。
3. 文案：明确“当前是 OC-Deploy 管理面板，点击进入 OpenClaw dashboard”。

## 5. 人机功效设计原则（执行层）

### 5.1 认知负担控制

1. 单屏只放一个主要任务。
2. 默认折叠低频高级配置。
3. 用“就地提示”替代长段说明。

### 5.2 错误预防优先于错误提示

1. 输入前校验（pattern、required、即时提示）。
2. 危险操作二次确认（删除、恢复原始配置、重启）。
3. 禁止无上下文错误文案（必须带对象和动作）。

### 5.3 状态可见性

1. 每个异步动作必须有开始、成功、失败三态反馈。
2. 按钮 loading 态与禁用态明确。
3. 保存/修改后自动刷新相关视图，减少“我改了但没生效”的疑惑。

### 5.4 一致性与可预测性

1. 同一动作词统一：添加、保存修改、删除、刷新。
2. 同级按钮顺序固定：次要在左，主要在右。
3. Toast 风格统一，禁止同类操作出现多套话术。

---

## 6. 文案与命名规范（去 Emoji）

### 6.1 全局命名

1. 主界面名称：`OC-Deploy 管理面板`
2. 外部入口名称：`OpenClaw 原生控制面板（dashboard）`
3. 避免使用：`控制台`（单独使用时语义不清）

### 6.2 按钮规则

1. 新建：`添加新模型` / `添加消息渠道`
2. 编辑：`保存修改`
3. 危险：`删除模型` / `删除渠道`
4. 系统动作：`启动 Gateway` / `停止 Gateway` / `重启 Gateway`

### 6.3 Toast 规则

1. Info：`正在执行...`
2. Success：`执行成功`
3. Warning：`已保存，但存在警告`
4. Error：`动作失败: 原因`

禁止：

1. 无动作主体：如“失败了”
2. 歧义主体：如“保存失败”（需标注保存什么）

### 6.4 去 Emoji 规则

1. 标题、按钮、Toast、说明文案默认不带 emoji。
2. 需图形引导时用统一 SVG 图标集（线性、1.5px）。
3. 禁止混用 emoji 与图标。

---

## 7. 视觉系统规范（可工程化）

### 7.1 设计 Token

1. 颜色 token：主色、功能色、文本色、背景色、边框色。
2. 间距 token：8px 基线（8/12/16/24/32）。
3. 圆角 token：`4/8/12` 三档。
4. 阴影 token：`sm/md` 两档，避免多档混乱。

### 7.2 排版层级

1. H1：页面级标题
2. H2：卡片级标题
3. H3：分组级标题
4. Body：14-16px
5. Meta：12-13px
6. Code：等宽字体

### 7.3 组件行为规范

1. Button：hover/focus/disabled 状态一致。
2. Card：标题、内容、操作区结构固定。
3. Form：label、required、error、hint 位置一致。
4. Badge：运行态、停止态、警告态统一样式体系。
5. Tooltip：必须使用现有“边界安全定位”实现。

---

## 8. 可访问性标准（必须验收）

目标：满足 WCAG 2.1 AA 的关键条款（本项目覆盖重点条目）。

### 8.1 可感知

1. 正文对比度 >= 4.5:1。
2. 大字/粗体对比度 >= 3:1。
3. 状态颜色不能作为唯一信息载体（需文字）。

### 8.2 可操作

1. 所有关键操作支持键盘 Tab/Enter/Escape。
2. 焦点可见（至少 2px 高对比边框/阴影）。
3. Tooltip 可通过键盘触发与关闭。

### 8.3 可理解

1. 表单错误提示必须指出字段与原因。
2. 危险操作弹窗必须说明后果。
3. 名称和术语全局一致。

### 8.4 健壮性

1. 不依赖按钮文字做逻辑判断。
2. 事件绑定需容错（元素不存在时不抛错）。

## 12. 发布与回滚策略

### 12.1 发布策略

1. 按阶段合并，不做巨型单提交。
2. 每阶段先在本地手测，再进入测试包。
3. 阶段之间保留可用状态（不可出现“中间态不可用”）。

### 12.2 回滚策略

1. 每阶段独立 commit，支持按阶段回退。
2. 若出现阻断问题，优先回滚当前阶段，不连带回滚已验证阶段。
3. 回滚后保留问题复盘记录，避免重复踩坑。

规则：

1. 单 commit 单主题。
2. 说明影响范围。

## 15. 给后续 AI 的执行指令模板

执行每个阶段时请遵循：

1. 先说明本阶段目标与改动文件。
2. 用 `rg` 定位目标代码，再最小化改动。
3. 修改后执行语法检查：`node --check app/ui/assets/management.js`。
4. 输出 diff 摘要与回归结果。
5. 等待确认后再进入下一阶段。

禁止事项：

1. 一次跨多个阶段大改。
2. 未验证就提交。
3. 修改业务逻辑时顺便重写 UI（必须拆开）。
